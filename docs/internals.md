# Internals

## Workspace Packages

- `packages/schemas`: Zod schemas and TypeScript types shared by core consumers and the API.
- `packages/core`: runtime-agnostic game-state creation, command application, events, IDs, and debug views.
- `apps/api`: optional Hono HTTP server, in-memory store, and event bus built around the core package.

Core is a TypeScript state library, not a backend. Hosts can call `createGame`, `applyCommand`, and `toGameView` directly in a browser, server, worker, test, or API wrapper.

## State Model

Core tracks one game state with ID `single`. API game IDs are host-level metadata and do not change the core state ID.

Game state contains:

- `revision`: incremented after every applied command.
- `players`: player ID, name, life, counters, and player zones.
- `zones`: shared battlefield, stack, exile, and command zones.
- `activePlayerId`, `priorityPlayerId`, `turnNumber`, `phase`, and `step`.
- `eventLog`: events generated by applied commands.

Player zones are `library`, `hand`, and `graveyard`. Shared zones are `battlefield`, `stack`, `exile`, and `command`.

## Objects

Objects represent cards, tokens, copies, spells, and abilities.

Important fields:

- `objectId`: identity of this object in its current zone.
- `cardId`: identity of the underlying represented card, when there is one.
- `kind`: `card`, `token`, `copy`, `spell`, `activatedAbility`, or `triggeredAbility`.
- `ownerPlayerId` and `controllerPlayerId`: optional player references.
- `counters`, `status`, `annotations`, and optional `visibility`.

When an object changes zones, the engine creates a new `objectId`, clears counters, resets status, and clears visibility unless command overrides apply. The `cardId` is preserved for represented cards.

Moving within the same zone keeps object metadata. Library reorder and shuffle clear visibility for objects in that library.

## Events and Revisions

Each command produces one event with:

- generated event ID,
- revision,
- timestamp,
- command type,
- optional actor player ID,
- human-readable message.

The API stores events inside the current game state's `eventLog` and publishes command events to current SSE subscribers.

## Views

`toGameView` currently returns a structured clone of the full state with `viewMode: "debug"`.

This means views expose all zones and hidden information. Add private or player-specific views before using this API for untrusted clients.

## API Store

`GameStore` is an optional in-memory host with multiple current game records.

- Server restart loses all game records.
- `POST /games` creates a named, empty game record with a generated API game ID.
- Commands mutate one stored game by replacing that record's state with the reducer result.
- API game metadata includes `id`, `name`, `createdAt`, and `updatedAt`.

## Rules Reference

The `rules/` directory contains Markdown rule reference files organized by Comprehensive Rules sections and glossary letters. These files are reference material for rules research, not executable engine behavior.
